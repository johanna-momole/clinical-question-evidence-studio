# Synthetic Data Methodology

## Scope and disclosure

Every patient record used anywhere in this application is **synthetic (fictional)**. No real patient data, real clinical records, or real-world health information is loaded, stored, or processed at any point. This is enforced structurally, not just by convention:

- `CohortSummary.is_synthetic` and `CohortProvenance.is_synthetic` are `Literal[True]` fields — they cannot be set to anything else without breaking type validation.
- Every API response that touches cohort data carries an explicit `synthetic_data_notice` string.
- Every Streamlit page that displays patient-level statistics opens with a synthetic-data banner.

This application has **not been clinically validated** and has **not been deployed in any healthcare organization**. It is a portfolio demonstration of clinical informatics engineering patterns only.

## Source of the synthetic FHIR data

The bundled dataset at `data/fixtures/fhir/synthetic_cohort_v1/` contains 160 synthetic patients represented as FHIR R4 resources (`Patient`, `Condition`, `Encounter`, `Observation`, `MedicationRequest`/`MedicationStatement`, `Procedure`), generated by a deterministic, seeded random process (seed=42) designed to resemble the kind of population a Synthea-style generator would produce for a T2DM/CKD cohort study. The generator and its parameters are not redistributed real-world data — they construct fictional patients, fictional encounter histories, and fictional lab values from scratch.

Each resource file also carries a `dataset_info.json` manifest documenting the generation seed and resource counts, which is read and surfaced via `GET /fhir/datasets` so that anyone inspecting the application can independently confirm the dataset's synthetic, seeded origin without having to trust prose claims alone.

## Loading pipeline

1. **Load** (`src/fhir/loader.py`) — reads every `.json` file in a dataset directory, skips `dataset_info.json`, and parses each FHIR resource. Malformed JSON, resources missing an `id`, unsupported resource types, and duplicate resource IDs across files are all captured as `FHIRIngestionError` records rather than silently dropped or silently overwritten.
2. **Parse** (`src/fhir/parser.py`) — converts each raw FHIR resource dict into a typed dataclass/Pydantic object (`parse_patient`, `parse_condition`, `parse_observation`, `parse_medication_request`, etc.), extracting the patient reference, coded value, and relevant dates. Resources that cannot be associated with a patient (e.g., a `Condition` with no `subject.reference`) are skipped (return `None`) rather than guessed at.
3. **Normalize** (`src/fhir/normalizer.py`) — assembles all parsed resources into a single in-memory `NormalizedDataset`, deduplicated by resource ID, with a `patient_ids` frozenset and `resource_type_counts()` summary.
4. **QA** (`src/qa/fhir_checks.py`) — runs ten structural checks (orphaned references, impossible dates, missing values, zero-encounter patients, etc.) against the normalized dataset before it is handed to the cohort engine. See [qa_framework.md](qa_framework.md) for the full check table.
5. **Store** — the normalized dataset is held in an in-process DuckDB analytical store (`src/fhir/service.py`), re-ingestable idempotently (re-running ingestion for a `dataset_id` replaces all prior rows rather than appending).

## Offline-only operation

All FHIR loading in Phase 3 operates exclusively on the bundled fixture files. There is no network call, no external FHIR server connection, and no live API in this pipeline. This was a deliberate scope boundary so the cohort construction step can run fully offline and reproducibly in any environment, including CI.

## What this does and does not establish

- It **does** demonstrate a realistic FHIR R4 ingestion → normalization → QA → analytical-store pipeline pattern that generalizes to real EHR-derived FHIR exports.
- It does **not** demonstrate performance, scale, or correctness against a real production FHIR server, a real Synthea installation, or real patient data of any kind.
- Any numeric result quoted from this pipeline (patient counts, prevalence percentages, attrition counts) describes the *fictional synthetic dataset only* and has no epidemiological meaning.
